# Engine Decision: Keep String (Candidate C)

The post-M0 engine decision evaluated three document model candidates against the FEAT-006 baseline: (A) a piece table with a `BTreeMap<usize, Piece>` over original and add buffers, (B) a rope via the `ropey` crate, and (C) the existing `String`-backed `Document` as a control. Each candidate was measured 5 times in release mode on the same machine (Apple M2, 16 GB RAM), using the identical measurement script from FEAT-006. Candidate C (String) matched the baseline within measurement noise on every metric. Candidate A (piece table) was comparable on open and startup but showed a ~8% regression on open_10k_ms and ~27% on save_ms, with 190 added lines of non-trivial BTreeMap splitting logic. Candidate B (rope) regressed open_10k_ms by 191% (0.2620 ms vs 0.0901 ms baseline), memory by 35% (3.41 MB vs 2.53 MB), and added an external dependency. Neither A nor B demonstrably beat C on any metric that matters for the product scope — all operations are sub-millisecond at realistic document sizes, and the product targets typical markdown files, not multi-megabyte logs. Per the decision criterion ("does not regress any baseline metric by more than 10% AND has the lowest complexity cost; default to C unless A or B demonstrably beats it"), the winner is **Candidate C: keep `String`**. Zero added lines, zero new dependencies, zero new unsafe, zero platform-specific code. The door remains open to revisit if a future feature (e.g., collaborative editing or extremely large file handling) demands a structural upgrade.
